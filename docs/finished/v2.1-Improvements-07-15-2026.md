# PR #33 — Release v2.1 AI ESS + Tesla Fleet Telemetry & EV Control Hardening

Branch: `v2-roadmap-iteration-2` → `main`
URL: https://github.com/JoshuaDodds/cerbomoticzgx/pull/33

This document merges the original PR #33 description (AI ESS daily-settlement
optimizer, weather shadow mode, PV nowcast, dashboard/advisor rework — already
live on GitHub, dated 2026-06-30) with the Tesla Fleet Telemetry + EV control
work staged since then. Nothing from the original description was dropped;

## Executive summary

This branch turns the AI ESS work into a fuller operator system and moves the
Tesla integration onto a cheaper, more correct control path:

- The optimizer is now **best-daily-settlement first**. It tries to maximize
  today's profit, or reduce today's cost as close to zero as possible, and
  only accepts a worse today for tomorrow when the future win is clearly
  exceptional relative to learned history and forecast risk.
- Forced grid charging is now **path-economic**. The old hard grid-charge
  price cap tunables were removed; the optimizer decides from buy/sell
  prices, PV/load forecasts, battery wear, stored-energy cost basis, and
  available future spreads.
- The dashboard is now an **operator dashboard**, not just a visibility page:
  Overview entry point, ESS tabs, guarded `.env` editing, replan requests,
  Import Schedule clearing, Trends, Weather, persistent Advisor chat,
  desktop/mobile navigation, external Battery/Venus views, and a manual
  **EV Start/Stop** control.
- Weather forecasting has a first implementation in **shadow mode** using
  keyless Open-Meteo. It logs/visualizes weather-derived HVAC and PV
  adjustments without changing control unless apply gates are deliberately
  enabled after validation.
- Same-day PV forecasting now has a **measured nowcast correction**. VRM
  remains the raw source forecast, but the optimizer raises today's near-term
  PV slots from live PV power and the latest settled PV slot when measured
  production proves VRM's remaining value is too low.
- Forecast and history tooling now supports actual-vs-forecast analysis for
  PV/load, projected-today profit, Advisor retrieval, richer charts, runtime
  control toggles, and regression tests.
- **Tesla integration moves from cost-metered REST polling to Fleet Telemetry
  streaming** (feature-gated behind `TESLA_TELEMETRY_ENABLED`, default off).
  The car pushes state via mTLS to an in-cluster receiver instead of being
  polled, cutting billable API reads/wakes to near zero.
- **EV charging is decoupled from the house-battery grid-assist toggle.**
  Charging intent now follows a dedicated `ev_charge_requested` flag with its
  own manual Start/Stop control; grid-assist only ever affects the house
  battery.
- A focused **audit of the EV/Tesla/telemetry control surface**
  (`docs/AUDIT_FINDINGS_2026-07-14.md`) found and fixed four High-severity
  control-correctness bugs (false-stop reporting, duplicate topic writers,
  grid-assist clobbering an AI BUY slot, an EV-intent self-latch) plus several
  Medium/Low items.

## Optimizer and control behavior

### Daily settlement policy

`lib/ai_powered_ess.py` now compares:

- a full-horizon plan across all available Tibber prices, and
- a today-first plan that protects the current 24-hour settlement result.

The selected policy is exported in `planning_policy` inside the plan JSON.
This prevents small forecasted gains tomorrow from causing a large known loss
today, while still allowing a future carryover when the price spike or
expected upside is large enough to be worth the forecast risk.

### Economic grid charging

The branch removes obsolete user-facing grid-charge price caps:

- `ESS_MAX_GRID_CHARGE_PRICE`
- `ESS_GRID_CHARGE_CHEAP_PCT`

Grid charging is now selected by modeled path economics. The remaining
`ESS_MAX_GRID_CHARGE_SOC` setting means only this: forced grid charging will
not target above that SoC, but PV surplus is still allowed to charge the
battery above that cap naturally.

### Cost basis, wear, and anti-churn guards

The optimizer now considers:

- battery cycle cost (`ESS_BATTERY_CYCLE_COST`),
- arbitrage margin (`ESS_ARBITRAGE_MARGIN`),
- persisted ESS cost basis (`ESS_COST_BASIS_PATH`),
- sell floor (`ESS_MIN_SELL_PRICE`),
- SELL hysteresis (`ESS_SELL_MIN_DWELL_MIN`, `ESS_SELL_HYSTERESIS_EUR`), and
- seasonal SoC reserve.

This reduces fragile thin-spread cycling and prevents active export below the
effective cost of stored grid energy.

### Grid-charge reporting and Victron slots

`ESS_MODEL_CHARGE_RATE` keeps settlement/reporting aligned with how Victron
scheduled charging behaves: full-power-to-target, then hold. Victron charge
slots are still derived from the optimizer trajectory; this is a
reporting/economics correction, not an unsafe direct control expansion.

### Manual grid-assist no longer touches the car (new)

`manage_grid_usage_based_on_current_price` in `lib/energy_broker.py` now
stands down during an AI `BUY` slot instead of applying a retain "cover-load"
setpoint that clobbered the optimizer's larger charge setpoint — a charging
battery is already retained, so nothing further is needed (audit **H3**).

Separately, and more fundamentally: **`grid_charging_enabled` (the dashboard
Grid Assist toggle) is now house-battery-only.** Previously toggling it also
started/stopped the vehicle; that coupling is removed. EV charging is driven
exclusively by the new `ev_charge_requested` intent (see below), so grid-assist
on/off can no longer race with, self-latch, or override vehicle charge state
(audit **H4**).

## Stop-command reliability hardening (found + fixed during PR review)

A follow-up adversarial review of this diff (post-audit, pre-merge) surfaced two more
issues in the stop-charge path, both fixed here:

- **Stale-status false negative.** `_stop_charge()`'s "nothing to drawing, nothing to stop"
  fast path trusted `tesla.is_charging == False` even when that reading was stale or had
  never been confirmed (e.g. a forced status refresh that failed right before the stop
  attempt, or the `False` default before the first read completes). Combined with the local
  meter's known ~3× under-read (audit M1) — which can put a genuine low-current taper below
  the 1A threshold — both signals could agree on "nothing to stop" while the car was still
  actually charging, silently defeating the original H1 fix under exactly the conditions it
  was meant to survive. Fixed with `_status_confirmed_not_charging()`: `is_charging == False`
  is only trusted when `tesla.last_update_ts` is within `STALE_STATUS_MAX_AGE_S` (5 min);
  otherwise the status is treated as unknown and the stop is attempted anyway (worst case: one
  avoidable wake; best case: catches a car that would otherwise have kept charging
  indefinitely).
- **Unbounded critical-spend retry loop.** The `critical=True` budget bypass added for
  `stop_charge_robust()` (so a stop can never be blocked by the spend guard) had no retry cap
  of its own — only the fixed 120s backoff. An unreachable car retrying a failed stop could
  spend past both the coded $9 monthly ceiling and Tesla's $10 free credit in under a day.
  Fixed: stop attempts are now bounded to `STOP_MAX_RETRIES` (5). Each attempt logs whether
  the failure was billable or not (`'network'` from `stop_charge_robust()` means a >=500
  response or transport exception — refunded, non-billable; anything else means Tesla
  accepted and billed the rejected command). Once exhausted, automatic retrying stops and a
  **critical-priority Pushover alert** (`pushover_notification_critical`, already used
  elsewhere for critical alerts) tells the operator to intervene manually — physically
  unplugging the car or stopping the charge in the Tesla app. A fresh, deliberate
  intent-off edge (the user explicitly requesting a stop again) resets the bounded-attempt
  counter rather than staying silently suppressed by an old escalation.

## Remaining audit follow-ups closed out (found + fixed during PR review)

Four more items from `docs/AUDIT_FINDINGS_2026-07-14.md`'s original backlog are now fixed,
plus one investigated and confirmed clean:

- **M3 — pre-command billable read in telemetry mode.** `wake_vehicle()` always spent a
  gated `data` read to check "online" before a command, even in telemetry mode where a
  streaming car is online by definition. Fixed with `_telemetry_considers_online()`: a
  telemetry refresh within `TELEMETRY_ONLINE_MAX_AGE_S` (5 min) is treated as proof of
  online and skips the billable check; a stale/disconnected stream still falls back to the
  real check.
- **M4 — `is_home` unknown before the first Location streams.** The Vehicle tab's "At
  home" row simply disappeared while `is_home` was `None`, giving no explanation for a
  manual Start press appearing to do nothing. Fixed in `app.js`: once other vehicle data is
  present, the row now shows "Unknown" instead of hiding, without adding any billable boot
  read (the cheaper of the audit's two suggested options).
- **L1 — misleading "waking vehicle" log in telemetry mode.** `EvCharger.main()`'s
  discovery-wake log always said "waking vehicle" even though `update_vehicle_status()`
  short-circuits to the pushed state and never wakes in telemetry mode. Fixed: the message
  now says "refreshing telemetry state" when telemetry is on. Added a shared
  `EvCharger._telemetry_on()` helper and reused it here and in the pre-existing
  `update_charging_amp_totals` mode check.
- **L2 — vestigial `charging_watts_reported` topic/state.** `ACChargingPower` translated
  to `Tesla/vehicle0/charging_watts_reported` (topic) and `tesla_ac_power_w` (state), both
  confirmed to have zero consumers anywhere in the codebase. Removed; `ACChargingPower` is
  now intentionally unmapped (returns `{}`) with a comment explaining why.
- **T2 — comment-drift sweep, confirmed clean.** A repo-wide grep for `poll`/`vehicle_data`
  outside the files this PR already touched found only unrelated hits (a subprocess
  `.poll()` check in `frontend/advisor.py`, and a Domoticz aux-data poll comment in
  `lib/helpers/startup_helpers.py`) — no further stale Tesla-polling comments to fix.

`docs/AUDIT_FINDINGS_2026-07-14.md` has been pruned again to drop these now-resolved items;
what's left there (M1 hardware CT scaling, L3 duplicate amp-totals logic, T1 stub file) is
the genuinely still-open backlog.

## Tesla API budget guard redesign (new)

`lib/tesla_budget.py` — the REST-call budget guard (`TESLA_BUDGET_*`) is
reworked from "worst-case-every-day" daily caps to a real hard **monthly**
ceiling plus a per-day runaway breaker:

- `MONTHLY_SAFETY_CEILING_USD` (unchanged, $9) is now the actual enforced
  guard, checked per-spend against the billing cycle's running total
  (`_spent_usd_month`). Charging is bursty (often a couple of days a week),
  so sizing daily caps as "worst case every single day" was too tight — it
  clamped caps down and could block a legitimate charge day while nowhere
  near the $10 credit.
- `DAILY_SAFETY_CEILING_USD` ($2) replaces the old daily-cap-clamped-to-
  monthly-worst-case math as a pure runaway circuit breaker (a stuck loop
  can't burn more than ~$1/day). Default daily caps raised accordingly
  (commands 40→300, data 50→150, wakes 6→20) since they're no longer the
  primary guard.
- **`TeslaBudget.spend(..., critical=True)`** — marks a call as
  safety-essential (a `charge_stop` and the wake needed to deliver it) so it
  is **never blocked** by either guard, only recorded. Leaving the car
  charging because a stop command hit a budget cap is worse than a fraction
  of a cent of overage. Previously a stop call could in principle be
  throttled by the same caps as routine polling — this closes that gap.
- **`TeslaBudget.refund(category, n)`** — reverses a previously-recorded
  spend when the call turns out non-billable (Tesla only bills responses
  `< 500`; a 5xx or network exception isn't billed, but we spend up-front to
  enforce the cap before knowing the outcome), keeping displayed usage in
  line with the Tesla portal.

## Bug fixes from the EV/Tesla/telemetry audit

A focused code audit of the EV/Tesla/telemetry/energy-broker surface found
these issues; all are fixed in this diff. `docs/AUDIT_FINDINGS_2026-07-14.md`
has been pared down to just the still-open, lower-priority follow-ups (it no
longer carries writeups for the items below, since those are now resolved —
this section is the record of what they were).

- **H1** — stop command falsely reported "stopped" while the car kept
  charging (could mask ongoing battery drain). Reverted to wake+retry
  escalation with the local meter as arbiter.
- **H2** — `event_handler` double-writing EV MQTT topics, causing
  amps/watts to flap between the telemetry value and a mis-scaled local
  value.
- **H3** — manual grid-assist clobbering the AI optimizer's BUY setpoint.
- **H4** — EV intent self-latch (`start_tesla_charge()` reading back its own
  write) that kept re-starting charge even after grid-assist was toggled
  off.
- **H6** — a config-schema type mismatch (`MINIMUM_ESS_SOC` as float, read
  with `int(...)` at import) crashed the Victron MQTT client thread at boot
  when the dashboard wrote `95.0` instead of `95`; fixed with tolerant
  `int(float(...))` parsing.
- **M1** (partial) — local evcharger amps under-read ~3×, causing perpetual
  amp re-issue and wasted API spend; the control-side fix (compare to last
  commanded amps, not the meter) is in this diff — the hardware-side CT
  scaling fix is still open.
- **M2** — `update_mqtt_and_domoticz` double-published vehicle topics on
  every command, competing with the bridge; MQTT writes now gated off in
  telemetry mode.
- **M5** — `tesla_is_plugged` was set from two disagreeing fields
  (`DetailedChargeState` and `ChargePortLatch`), which could flap
  True/False mid-charge; `ChargePortLatch` is now confirm-only.

**Still open (not blocking, tracked in the audit doc):** M3 (skip the
pre-command online read in telemetry mode), M4 (seed `is_home` on boot before
the first Location streams), plus L1–L3 and T1–T2 cosmetic/cleanup items.

## Forecasting and analytics

### Weather shadow mode

New file: `lib/weather.py`.

Weather integration:

- fetches Open-Meteo without an API key,
- uses `HOME_ADDRESS_LAT` / `HOME_ADDRESS_LONG` from `.secrets`,
- caches latest forecast data under `data/weather/latest.json`,
- appends compact summaries to `data/weather/weather.ndjson`,
- computes symmetric heating/cooling degree-day load adjustments, and
- computes GTI-shaped PV shadow forecasts.

Default apply gates remain off:

- `HVAC_LOAD_APPLY=False`
- `PV_WEATHER_APPLY=False`

The intent is to compare weather-adjusted forecasts against actual PV/load
for 1-2 weeks before allowing them to affect a 16 kW controller.

### PV nowcast and VRM forecast reporting

Files changed:

- `lib/energy_broker.py`
- `frontend/data.py`
- `frontend/static/js/app.js`
- `tests/test_energy_broker.py`
- `tests/test_90_mobile_ux_static.py`

Why this changed:

The raw VRM solar forecast can be correct as a full-day total but wrong as a
remaining-day control input late in the day. On 2026-06-29 the system was
still producing meaningful PV while VRM's `pv_projected_remaining` had
collapsed near zero. That made the optimizer schedule too little PV in the
remaining slots, and the Overview Solar card contradicted the schedule once
the optimizer was corrected.

Implementation notes:

- `_latest_settled_pv_slot_kwh(slot_duration_h, now)` reads today's newest
  complete settlement row, rejects stale/incomplete rows, checks slot
  duration, and normalizes actual PV to the optimizer slot size.
- `_pv_nowcast_anchor_kwh(slot_duration_h, now)` blends the latest settled
  slot with live `STATE.get("pv_power")`. If live PV has dropped materially
  below the recent settled slot, it treats that as sunset/tree-line
  drop-off and lowers confidence instead of blindly extrapolating the prior
  high output.
- `_apply_pv_nowcast(pv_forecast, forecast_slots, weather_context,
  slot_duration_h, now)` raises only the current day's near-term PV
  forecast, fades the correction over the next few hours, weights the fade
  by Open-Meteo GTI ratios when available, and leaves tomorrow unchanged.
  It annotates `weather_context["summary"]` with `pv_nowcast_*` fields for
  inspection.
- `_publish_plan_json()` now exports both the raw and adjusted values:
  `pv_remaining_raw_wh`, `pv_remaining_raw_source` (`"VRM forecast"`),
  `pv_adjusted_remaining_wh`, `pv_adjusted_remaining_source`, and
  `pv_adjustment_kwh`. The original `pv_remaining_wh` is retained for
  compatibility.
- `frontend/data.py` passes those fields through `/api/plan`.
- The Overview Solar card now shows the optimizer-adjusted remaining PV as
  the main value and explicitly labels the raw source underneath as
  `VRM forecast`. This avoids hiding provenance while making the card match
  the schedule the optimizer is actually using.

Important design choice:

This is deliberately not behind `PV_WEATHER_APPLY`. Weather PV shaping is a
forecast model that still needs validation. The nowcast correction is based
on measured production from the live system and recent settlement data, so
it is a control-time reconciliation layer rather than a speculative
weather-model apply gate.

## Removed or deprecated behavior

- `ESS_MAX_GRID_CHARGE_PRICE` and `ESS_GRID_CHARGE_CHEAP_PCT` are no longer
  active user knobs and should be removed from live `.env` files.
- The dashboard should no longer be described as visibility-only. It has
  guarded config/replan/restart/schedule-clear/EV-start-stop controls.
- The optimizer should no longer be described as monthly-settlement-first.
  It is now daily-settlement-first with learned exceptional-future
  exceptions.
- The raw solar forecast in UI text should be called `VRM forecast`; the
  adjusted schedule/card value should be called optimizer-adjusted or
  optimizer nowcast, not "raw forecast".
- User-facing action labels are `IDLE`, `RETAIN`, `BUY`, and `SELL`.
- **The Grid Assist toggle no longer implies "charge the car."** Any
  operator documentation/runbooks describing grid-assist as a vehicle
  charging mode are now wrong — it is house-battery-only. Vehicle charging
  is controlled solely by the new Start/Stop control / `ev_charge_requested`.
- The legacy `tesla_charge_requested` state key is repointed to
  `ev_charge_requested`.

## Rollout / risk

- Fleet Telemetry is off by default (`TESLA_TELEMETRY_ENABLED=False`) — the
  legacy REST path is unchanged and remains the default.
- One-time telemetry enrollment needs: `.well-known` reachable on
  `fleet.hs.mfis.net:443`, receiver on `:6443`, partner domain registered,
  config pushed via the proxy (`scripts/tesla_register_partner_domain.py`,
  `scripts/tesla_push_telemetry_config.py`). See the telemetry deploy notes
  in `deploy/tesla-fleet-telemetry/`.
- Telemetry cost: ~$1/150k signals (effectively free); the budget guard
  still hard-caps REST command/data/wake spend at the $10 credit regardless
  of mode.
- EV control decoupling changes operator-facing behavior: toggling Grid
  Assist will **no longer** start or stop the car — operators relying on
  that side effect need to use the new Start/Stop button instead.
